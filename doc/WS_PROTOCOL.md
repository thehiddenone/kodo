# Kodo — WebSocket Protocol

> Status: draft for review. Reference: [DESIGN.md](DESIGN.md), [REQUIREMENTS.md](REQUIREMENTS.md), [STATE_AND_LIFECYCLE.md](STATE_AND_LIFECYCLE.md).

This document specifies the WebSocket protocol between the VS Code extension (client) and the Kodo Server (server). It is the sole contract the two processes share; everything else is internal.

---

## 1. Purpose

The protocol exists for exactly two reasons:

1. **Visibility** — show the user what Kodo is doing right now.
2. **Interaction** — let Kodo and the user exchange information when one needs something from the other.

Every message in the catalogue (§5–§7) serves one of these. Anything that does not — internal session bookkeeping, workspace mechanics, mirror operations, scheduler decisions — stays off the wire and lives in session logs, the workspace event log, or the mirror, as defined in [STATE_AND_LIFECYCLE.md](STATE_AND_LIFECYCLE.md).

A second filter applies: messages encode events at the user's level of abstraction, not the engine's. The user thinks in terms of "an agent is working", "a file landed in my project", "Kodo asked me a question". They do not think in terms of session IDs, request_id dedup, or supersession chains.

---

## 2. Envelope

Every WebSocket frame is a single JSON object:

```json
{
  "kind": "request" | "response" | "event" | "stream_chunk" | "stream_end",
  "id": "<hex uuid>",
  "correlation_id": "<id of the request this frame answers or streams under>",
  "payload": { "type": "<message-type>", ... }
}
```

Fields:

- `kind` — frame discriminator. See §2.1.
- `id` — unique frame identifier, generated by the sender.
- `correlation_id` — present on `response`, `stream_chunk`, and `stream_end`. Absent on `request` and `event`.
- `payload` — message-specific body. The `payload.type` string is the sub-discriminator that identifies the concrete message.

### 2.1 Frame kinds

| Kind | Direction | Carries `correlation_id` | Purpose |
|---|---|---|---|
| `request` | C→S **or** S→C | no | Initiates a request that expects a `response`. |
| `response` | The opposite of the originating `request` | yes (= request's `id`) | Terminates a request. Carries either result fields or `payload.error`. |
| `event` | S→C only | no | Unsolicited push the client renders. No reply expected. |
| `stream_chunk` | S→C only | yes (= stream's `id`) | One fragment of a streamed payload. |
| `stream_end` | S→C only | yes (= stream's `id`) | Terminates a stream. |

**`kind=request` is bidirectional.** Server-initiated requests are how Kodo asks the user something (§6). The user's reply is a `kind=response` with `correlation_id` set to the server request's `id`. There is no "prompt_id" field — `id` and `correlation_id` already carry the relationship.

### 2.2 Errors

An error in response to a request is a `response` frame whose payload carries `error`:

```json
{ "type": "error", "error": { "code": "<short_code>", "message": "...", "recoverable": true } }
```

Unsolicited server errors are `event` frames with `payload.type = "error"` (§5.7).

### 2.3 Streams

A stream is a series of `stream_chunk` frames sharing one `correlation_id`, terminated by exactly one `stream_end` with the same `correlation_id`. The chunk payload's `type` is the same for every chunk in a stream and identifies what is being streamed (today: `agent.tokens`). Other event types do not stream in MVP.

The originating `id` of a stream is generated by the server when the stream begins; clients allocate per-stream buffers keyed on that id when the first chunk arrives.

## 3. Transport

- **Binding.** Loopback (`127.0.0.1`) only, on a port the extension picks at activation (FR-SRV-03/04).
- **Single connection.** The server accepts one client connection at a time; a new connection replaces the prior one (FR-VSIX-03 says one server per VS Code window; the extension is the sole client).
- **Keepalive.** WebSocket-level ping/pong is the responsibility of the WS library on both sides. The application protocol has no keepalive of its own.
- **Encoding.** UTF-8 text frames. Binary frames are not used.
- **Reconnect.** §8.

## 4. Connection lifecycle

### 4.1 Hello

After establishing the WebSocket, the client sends a `hello` request:

```json
{ "kind": "request", "id": "<id>",
  "payload": { "type": "hello", "client": "vsix", "client_version": "0.1.0" } }
```

The server replies with the current world:

```json
{ "kind": "response", "id": "<id>", "correlation_id": "<hello id>",
  "payload": {
    "type": "hello.ack",
    "server_version": "0.1.0",
    "project_root": "/abs/path",
    "state": { ...state snapshot per §5.1... }
  } }
```

The embedded `state` is what the panel needs to render its initial view; no separate `state` event is required after `hello.ack`. Subsequent state changes are pushed as `state` events (§5.1) when material.

### 4.2 Shutdown

Either side may close the WebSocket. The server closes it on graceful shutdown (FR-SRV-07). The client closes it when the VS Code window closes. Neither side needs to send a protocol-level "goodbye" frame; closing the WS is sufficient.

## 5. Visibility events (server → client)

All visibility messages are `kind=event` frames. The client renders them and never responds.

### 5.1 `state` — workflow snapshot

Pushed on connect (embedded in `hello.ack`), and whenever a field below changes in a way the panel must reflect (phase transition, agent change, autonomous toggle, etc.). Always carries the complete current snapshot — never a delta — so the panel can replace its cached state wholesale.

```json
{ "type": "state",
  "phase": "intake" | "running" | "awaiting_user" | "stopped" | "done",
  "current_agent": {
    "agent_id": "<id>",
    "name": "functional_designer",
    "model": "claude-sonnet-4-6",
    "target_responsibility_code": "TRADE",
    "target_type": "functional-design"
  } | null,
  "autonomous": false,
  "cumulative_usd": 1.42,
  "pending_prompts": [
    { "prompt_id": "<id>", "kind": "question" | "approval" | "permission", "summary": "..." }
  ],
  "last_checkpoint_sha": "abc123..." | null
}
```

`phase` semantics:

- `intake` — no narrative yet; user input via `prompt.submit` (§7.1) is being collected.
- `running` — an agent is active; cards stream into the panel.
- `awaiting_user` — Kodo is blocked on a `prompt.*` request the user has not yet answered.
- `stopped` — STOP was issued; no work is in flight.
- `done` — all components are code-complete and the final gate has been approved.

`pending_prompts` lets a freshly-connected panel know which server-initiated requests are still open so it can rehydrate the corresponding cards immediately.

### 5.2 `agent.started` / `agent.finished` — invocation boundaries

```json
{ "type": "agent.started",
  "agent_id": "<id>",
  "name": "test_designer",
  "model": "claude-sonnet-4-6",
  "target_responsibility_code": "AUTH",
  "target_type": "test-plan" }
```

```json
{ "type": "agent.finished",
  "agent_id": "<id>",
  "status": "completed" | "stopped" | "errored" }
```

`agent_id` is unique per invocation and ties together every event the agent produces (tokens, tool calls, finished). The same logical sub-agent name running again gets a new `agent_id`.

### 5.3 `agent.tokens` — streamed LLM output

Streamed via `stream_chunk` + `stream_end` (§2.3). Each chunk:

```json
{ "kind": "stream_chunk", "correlation_id": "<stream id>",
  "payload": { "type": "agent.tokens", "agent_id": "<id>", "text": "...delta..." } }
```

Terminated by `stream_end` with the same `correlation_id` and `payload: {"type": "agent.tokens"}`.

A single agent invocation may produce multiple streams (multiple LLM calls within one agent run); each gets its own stream id.

### 5.4 `agent.tool_call` — visible tool invocations

Surfaced only for tools the user benefits from seeing: shell, escalation, user dialog. **Workspace tools (`publish_artifact`, `read_artifact`) and the internal report tools are not surfaced as `agent.tool_call`** — their effects are conveyed by `artifact.published` (§5.6), `review.*` (§5.5), and the `prompt.*` requests (§6).

```json
{ "type": "agent.tool_call",
  "agent_id": "<id>",
  "tool_name": "tools/shell.run",
  "summary": "pytest -q  →  exit 0  (47 passed)",
  "details": { ... tool-specific payload ... } }
```

The `summary` is one line suitable for a card header. `details` carries tool-specific structured data the panel may expand on click. For `tools/shell.run`: `{command, cwd, exit_code, stdout, stderr}` (with stdout/stderr truncated by the server if large).

### 5.5 `review.started` / `review.verdict` — low-fi workspace activity

Workspace activity is intentionally low-fidelity in MVP: the user sees that critic loops are happening and roughly how they are going, but not the artifact content, diffs, or feedback bodies. These two events provide the rhythm:

```json
{ "type": "review.started",
  "reviewer_name": "functional_design_critic",
  "target_filename": "functional_design.md",
  "target_type": "functional-design" }
```

```json
{ "type": "review.verdict",
  "reviewer_name": "functional_design_critic",
  "target_filename": "functional_design.md",
  "verdict": "accepted" | "rejected",
  "concern_count": 3 }
```

`target_filename` is a label only — not a path the panel can open. Workspace paths and contents stay off the wire.

This set is deliberately minimal. §9 records the opt-in "verbose workspace" mode planned for post-MVP that would add fully-detailed workspace events; the MVP set is structured so that addition is purely additive.

### 5.6 `artifact.published` — Promoter landed a file

Fires when [STATE_AND_LIFECYCLE.md §8](STATE_AND_LIFECYCLE.md) Promoter completes a promotion: a new or revised file is now in `src/` or `gen/` and a checkpoint commit exists in the mirror.

```json
{ "type": "artifact.published",
  "artifact_id": "<uuid>",
  "artifact_type": "functional-design" | "code" | "test" | ...,
  "project_code": "ETRD",
  "responsibility_code": "TRADE",
  "path": "src/design/trade_execution/functional_design.md",
  "checkpoint_sha": "<sha>",
  "previous_checkpoint_sha": "<sha>" | null,
  "previous_path": "src/design/trade_execution/old_name.md" | null,
  "view": "full" | "diff" }
```

`view` semantics:

- `"full"` — first appearance of this artifact at this path. Panel opens `path` directly.
- `"diff"` — revision. Panel invokes `vscode.diff(prev, curr)` where the prior side is materialized by the extension from the mirror via `git show <previous_checkpoint_sha>:<previous_path ?? path>` against `<project>/.kodo/checkpoints/`.

`previous_path` is omitted when it equals `path` (no rename). When a revision renamed the file, `previous_path` is the prior committed path and the extension uses it to extract the prior version from the mirror.

The mirror is the canonical source for prior versions; no separate server endpoint exists to fetch them.

### 5.7 `artifact.removed` — Promoter deleted a file

Fires when an artifact is retired without a successor, causing Promoter to delete the corresponding file in `src/`/`gen/` and commit the deletion to the mirror. Rare in MVP (the critic loop never produces no-successor retirements; only user-driven workflow re-entry does, which is largely post-MVP), but specified now so the gap is closed.

```json
{ "type": "artifact.removed",
  "artifact_id": "<uuid>",
  "artifact_type": "...",
  "project_code": "ETRD",
  "responsibility_code": "TRADE",
  "path": "gen/src/trade_execution/old_module.py",
  "checkpoint_sha": "<sha>" }
```

### 5.8 `orchestrator.compacted` — Orchestrator session rotated

Fires when [STATE_AND_LIFECYCLE.md §4.5](STATE_AND_LIFECYCLE.md) compaction completes: the prior Orchestrator session was summarised, a fresh session was started with the compacted block as initial context, and work is resuming transparently.

```json
{ "type": "orchestrator.compacted",
  "from_session_id": "<uuid>",
  "to_session_id": "<uuid>",
  "summary_excerpt": "...first ~280 chars of the compacted prior-context block, for display..." }
```

The panel renders a low-fi banner ("Kodo compacted its memory at $X.YZ — work is continuing") so the user understands cost spikes and the slight conversational reset. No user action is required.

### 5.9 `usage.update` — cost accounting

Pushed after each LLM call.

```json
{ "type": "usage.update",
  "cumulative_usd": 1.42,
  "last_call": {
    "agent_name": "test_designer",
    "model": "claude-sonnet-4-6",
    "input_tokens": 8421,
    "output_tokens": 1203,
    "cache_read_tokens": 12044,
    "cache_write_tokens": 0,
    "usd": 0.043
  } }
```

### 5.10 `error` — unsolicited server error

For errors not tied to a specific request. Errors tied to a request are returned as `response` payloads with an `error` field (§2.2).

```json
{ "type": "error",
  "code": "llm_quota" | "llm_retrying" | "internal" | ...,
  "message": "...",
  "recoverable": true | false,
  "retry_after_seconds": 8 | null }
```

`recoverable: false` indicates the engine has halted and Dev intervention is required (auth failure, billing, NFR-04 violation).

## 6. Interaction prompts (server → client requests)

Every server-initiated user prompt is a `kind=request` frame. The client's reply is a `kind=response` whose `correlation_id` equals the request's `id`. There is no separate "prompt_id" namespace.

While a prompt is outstanding it also appears in `state.pending_prompts` (§5.1), so a freshly-connected panel can rehydrate cards for prompts the user has not yet answered.

### 6.1 `prompt.question` — free-form or choice

Used by the engine to ask the user a clarifying question. Covers any agent's `ask_user` call (e.g. Narrative Author or a critic eliciting or validating user-supplied input, or the Orchestrator's own judgment-call questions; not exposed as a distinct wire message), the intake "ready or more?" question, and any `escalate_blocker` the Orchestrator chooses to surface to the user as a written or chosen reply.

Request payload:

```json
{ "type": "prompt.question",
  "asker_name": "narrative_author" | "engine" | ...,
  "question": "Should the bot handle pre-market hours?",
  "mode": "free_text" | "choice",
  "choices": [ { "key": "yes", "label": "Yes" }, { "key": "no", "label": "No" } ] | null }
```

Response payload (`mode: "free_text"`):

```json
{ "type": "prompt.question.response", "answer_text": "..." }
```

Response payload (`mode: "choice"`):

```json
{ "type": "prompt.question.response", "choice_key": "yes" }
```

### 6.2 `prompt.approval` — review gate

Surfaced when a critic or solo agent calls `request_user_review_artifact` at an FR-WF-05 review gate, presenting a converged artifact for the user's sign-off. The two affirmative actions defined by FR-WF-06 (`agree` / `feedback`) are the only valid responses. In autonomous mode the gate is auto-accepted and no `prompt.approval` is emitted (FR-AUT-02).

Request payload:

```json
{ "type": "prompt.approval",
  "gate_type": "narrative" | "responsibilities" | "requirements" | "design" | "test_plan" | "implementation" | "final",
  "summary": "Narrative ready for your review.",
  "artifact_id": "<uuid>" | null,
  "artifact_path": "src/narrative/narrative.md" | null }
```

When `artifact_path` is present the panel renders an "Open" link. When `artifact_id` is present the panel can correlate against `artifact.published` events it has already shown.

Response payload:

```json
{ "type": "prompt.approval.response",
  "action": "agree" | "feedback",
  "feedback_text": "..." | null }
```

`feedback_text` is required when `action = "feedback"`. On `agree`, the reviewing agent reports the artifact completed (`report_artifact_completed`) and the engine promotes it; on `feedback`, the reviewing agent opens a revision round on the same artifact, escalating to the Orchestrator via `escalate_blocker` only when the feedback implicates an upstream artifact.

### 6.3 `prompt.permission` — security-layer prompt

Used when the security layer's rule evaluation yields `prompt` (FR-SEC-05) for a tool call the agent attempted.

Request payload:

```json
{ "type": "prompt.permission",
  "agent_id": "<id>",
  "tool_name": "tools/shell.run",
  "summary": "curl https://api.example.com/data",
  "matched_rule_reason": "shell command matches network-call prompt rule",
  "details": { ... tool-call arguments ... } }
```

Response payload:

```json
{ "type": "prompt.permission.response",
  "action": "allow" | "deny",
  "remember": "session" | "global" | "no" }
```

`remember` triggers the security layer to install a matching `allow`/`deny` rule at the requested scope (FR-SEC-03). `"no"` applies the decision only to this single call.

## 7. Client commands (client → server requests)

The client initiates these. Each is a `kind=request`; the server replies `kind=response` with an acknowledgement or the requested data.

### 7.1 `prompt.submit` — user-initiated input

The "user has something to say to Kodo" channel. Always valid; the engine interprets it in light of current state.

In MVP it drives the intake phase: the user types the project description (one or more `prompt.submit` calls), each followed by the engine asking "ready, or more?" via a `prompt.question` (§6.1). On `"ready"` the engine starts Narrative Author and the phase transitions to `running`.

Post-MVP the same wire message extends to "change a requirement", "tweak the implementation", and similar user-initiated re-entries on a finished project. The server-side handler grows; the wire shape does not change.

Request payload:

```json
{ "type": "prompt.submit", "text": "Build an algorithmic trading bot for E*TRADE that ..." }
```

Response payload:

```json
{ "type": "prompt.submit.ack" }
```

### 7.2 `stop` — global STOP

FR-VSIX-07. Cancels the worker, aborts in-flight LLM streams (FR-LLM-07), and transitions `state.phase` to `stopped`.

```json
{ "type": "stop" }
```

Response:

```json
{ "type": "stop.ack" }
```

A `state` event with `phase: "stopped"` follows the ack.

### 7.3 `checkpoint.list` — list mirror commits

```json
{ "type": "checkpoint.list" }
```

Response:

```json
{ "type": "checkpoint.list.response",
  "commits": [
    { "sha": "<sha>", "message": "ETRD/TRADE/code: <session> → <artifact>", "timestamp": "<iso8601>" }
  ] }
```

### 7.4 `checkpoint.rollback` — restore to a prior checkpoint

[STATE_AND_LIFECYCLE.md §8.3](STATE_AND_LIFECYCLE.md). Terminates all sub-agent sessions, clears the workspace, restores `src/`/`gen/` from the target commit, and rebuilds the index.

```json
{ "type": "checkpoint.rollback", "sha": "<target sha>" }
```

Response:

```json
{ "type": "checkpoint.rollback.response", "state": { ...state snapshot... } }
```

The embedded `state` reflects the post-rollback world. A `state` event with the same content is also pushed so other components observing event streams stay in sync.

### 7.5 `mode.set` — toggle autonomous mode

FR-AUT-01.

```json
{ "type": "mode.set", "autonomous": true }
```

Response:

```json
{ "type": "mode.set.ack", "autonomous": true }
```

A `state` event with the updated `autonomous` field follows.

### 7.7 `config.reload` — apply settings.json changes

Tells the server to re-read `settings.json` (both user and project layers) and apply whatever changed.  The primary use case is model switching: the VSIX edits the `models` dict in `<project>/.kodo/settings.json` (toggling `"active"` on the desired entry) and then sends this message.  The server resolves the new active plugin and replaces the engine's LLM plugin in-place.

The message carries no payload fields; all configuration comes from disk.

```json
{ "type": "config.reload" }
```

Response:

```json
{ "type": "config.reload.ack" }
```

The full `models` schema is documented in [SETTINGS.md](SETTINGS.md) §2.2.

**Model registration** is equivalent to adding an entry to this dict and sending `config.reload`.  No separate registration message exists.

### 7.8 `credentials.set` — deliver an API key

Sends a vendor API key to the server.  Keys are held in memory only and never written to disk.  The VSIX stores keys in VS Code SecretStorage and delivers them over the loopback WebSocket immediately after `hello.ack`.  Sending a key for a vendor that already has one replaces the prior value.

After storing the key the server resolves the active plugin (same logic as `config.reload`) so that a freshly delivered Anthropic key takes effect immediately without a separate `config.reload`.

```json
{ "type": "credentials.set", "vendor": "anthropic", "api_key": "sk-ant-..." }
```

Supported vendors: `"anthropic"`, `"huggingface"`.  Additional vendors are added as new plugins are introduced.

Response:

```json
{ "type": "credentials.set.ack" }
```

### 7.6 `security.add_rule` — install a security rule

FR-SEC-07.

```json
{ "type": "security.add_rule",
  "scope": "session" | "global",
  "rule": {
    "match": { "tool": "tools/shell.run", "args_regex": "^pytest( |$)" },
    "action": "allow" | "deny" | "prompt",
    "reason": "free-form, displayed to Dev"
  } }
```

Response:

```json
{ "type": "security.add_rule.ack" }
```

## 8. Reconnect semantics

The server's outbox (`src/kodo/transport/_outbox.py`) buffers events while the client is disconnected. On reconnect:

1. The server replays every buffered envelope in arrival order over the new WebSocket, **before** processing any new client frames.
2. After replay, the server pushes a fresh `state` event so the client can reconcile if anything was dropped at the outbox cap (`50 MB` default; overflow drops the oldest frames and logs an error — best-effort per FR-WS-04).
3. The client may now send its `hello` request. The response embeds the same `state` snapshot, redundantly, so first-connect and reconnect follow identical client logic.

Pending server-initiated prompts (§6) survive disconnect: the original request frame is in the outbox and replayed; `state.pending_prompts` also lists them. The client renders whichever it sees first; the duplicate is idempotent because both carry the same `correlation_id`.

## 9. Non-goals

The following are deliberately **not** on the wire. Future contributors should not add them without a design conversation.

- **Session log content.** The audit trail (STATE_AND_LIFECYCLE.md §5) lives on disk; the protocol carries no message that exposes session JSONL.
- **Workspace artifact content, paths, or diffs (MVP).** Workspace activity is conveyed only by `review.*` (§5.5). Inspection of in-flight artifacts is a planned post-MVP feature (see "Verbose workspace mode" below), gated behind a settings flag and shipped as additive event types so the MVP catalogue stays unchanged.
- **Mirror operation internals.** Mirror commits surface only as `checkpoint_sha` references on `artifact.published`/`artifact.removed`. Commit objects, trees, and refs are not exposed.
- **Stage-machine transitions.** The workflow's internal stages are not pushed as their own events. The user-visible signal is the sequence of `agent.started`/`agent.finished` events plus `state.phase`.
- **Memory writes (`src/.memory/*.md`) as a distinct event class.** They flow through `artifact.published` like every other promoted artifact.
- **Historical event replay beyond the outbox.** The protocol is "now and forward". The on-disk workspace event log, session logs, and mirror are the archives; the panel is a live view, not a history browser.
- **Keepalive at the application layer.** WebSocket ping/pong handles liveness.
- **Backpressure / flow control.** Out of scope per FR-WS-04. The outbox cap is the only bound.

### 9.1 Planned additive extensions (post-MVP)

These are anticipated and the protocol is structured so adding them is purely additive — no existing message type changes shape.

- **Verbose workspace mode.** New event types `workspace.artifact.published`, `workspace.artifact.retired`, `workspace.feedback.published` etc., gated by a `workspace.verbose` settings flag. Carry workspace paths and structured content references. The existing `review.*` events stay as the low-fi default.
- **User-initiated workflow re-entry.** `prompt.submit` already covers the wire side; server-side routing expands without changing the message shape.
- **Streamed tool output.** A `stream_chunk` form of `agent.tool_call` for long-running shell commands. Existing single-event form remains valid for short calls.
